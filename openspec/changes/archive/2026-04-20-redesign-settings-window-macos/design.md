## Context

当前 Settings 页面集中在 `src/pages/Settings/index.tsx` 与 `src/pages/Settings/index.css` 中，采用顶部 tabs 切换 `general`、`agents`、`data` 三个分区。功能已覆盖语言切换、监控模式、Agent 开关、保留天数和数据清理，但页面主要由原生表单控件与局部样式拼接而成，视觉层级偏平，分组关系不够清晰。

Settings 窗口由 `src-tauri/src/lib.rs` 创建，当前使用标准 Tauri 窗口配置并单独打开 `/settings` 路由。这个窗口已经具备独立设置窗口的基础，但页面内容的表单观感、暗色模式和危险操作表达，与 macOS 偏好设置窗口仍有明显差距。

本次变更聚焦“更像 macOS 的设置窗口体验”，但仍遵循仓库“如无必要，勿增实体”的约束，优先通过现有页面结构、CSS 和少量窗口配置微调完成，不引入新的 UI 组件库。

## Goals / Non-Goals

**Goals:**
- 明确 Settings 窗口应采用的 macOS 风格视觉层级，并把要求落到可实现的页面结构上。
- 保留现有 General / Agents / Data 信息架构，重点优化导航、分组卡片、控件语义、留白和亮暗色表现。
- 让窗口层与内容层的体验更加一致，避免内容区像普通网页嵌入到原生窗口里。
- 在不引入新依赖的前提下，为后续实现提供清晰边界和任务拆分。

**Non-Goals:**
- 不改变 Settings 页面已有功能语义，不新增新的设置项或后端 command。
- 不引入完整的自定义窗口框架或完全仿原生 AppKit 控件。
- 不把三段式顶部导航改造成全新的 sidebar 信息架构。
- 不在本次变更中处理与 Settings 无关的全局设计系统重构。

## Decisions

### 1. 保留顶部导航，不切换为侧边栏
- **Decision**: 继续使用 `General / Agents / Data` 顶部导航，但将其收敛为更接近 macOS segmented control 的表现：居中、非全宽、激活态更轻、与内容卡片形成上下层级。
- **Why**: 当前只有三个一级分区，信息量不足以支撑 sidebar。保留现有 IA 可以减少结构改动，集中投入在视觉质量提升上。
- **Alternatives considered**:
  - **改成左侧 sidebar**：更像现代 System Settings，但对当前三类内容来说过重，且会引入更多布局调整。
  - **完全取消导航改为单页滚动**：实现简单，但会削弱设置项分组识别度。

### 2. 内容区采用 grouped sections，而不是裸露表单行
- **Decision**: 每个 tab 内容都组织成一到多个 inset grouped card；卡片内部使用分隔线、统一行高、主次文字层级和标准化操作区。
- **Why**: 这能最快建立 macOS 偏好设置窗口的视觉识别，同时保持现有数据结构不变。
- **Alternatives considered**:
  - **维持当前裸行布局**：成本低，但无法显著改善页面气质。
  - **改成大量独立卡片**：会让页面变碎，不符合系统偏好设置的收敛感。

### 3. 使用现有技术栈完成原生感增强，不新增组件库
- **Decision**: 继续使用 React + 本地 CSS + Framer Motion，通过结构微调和样式重写实现 macOS 风格；不引入额外 UI 库。
- **Why**: 当前页面范围小、控件种类有限，引入新库会增加风格混杂与维护成本，也违背最小改动原则。
- **Alternatives considered**:
  - **引入 Headless / Radix 组件**：控件能力更强，但当前需求主要是视觉统一，收益不足以覆盖引入成本。

### 4. 保留原生窗口装饰，优化窗口与内容的协调感
- **Decision**: 保留 settings 窗口的原生装饰和系统控制区，不走自定义无边框窗口路线；如有需要仅调整初始尺寸、最小尺寸或内容内边距来改善整体比例。
- **Why**: 原生标题栏本身就是 macOS 语义的一部分。当前违和感主要来自内容，而非必须重做窗口 chrome。
- **Alternatives considered**:
  - **自定义 titlebar / transparent window / vibrancy 深度改造**：理论上更接近原生材质，但实现和回归成本较高，不适合作为这次优化的首选路径。

### 5. 危险操作采用“语义收敛”而不是“高饱和警告块”
- **Decision**: Data 区保留 `清理旧数据` 与 `清空全部数据`，但危险操作在视觉上应通过红色文字、次级按钮和确认层级来表达，而不是大面积红底。
- **Why**: macOS 偏好设置通常把 destructive intent 放在文案和交互层级，而不是用强烈警告面板主导页面。
- **Alternatives considered**:
  - **继续使用红底红框按钮和确认块**：实现简单，但 Web 感明显，且会破坏页面整体克制感。

## Risks / Trade-offs

- **[视觉目标较主观]** → 通过 specs 把“导航位置、分组形式、危险操作表达、亮暗色层级”写成可验证要求，避免只停留在“更好看”。
- **[单文件 Settings 页面后续实现易变复杂]** → 任务中明确先做结构梳理，再做样式与窗口微调，避免在一个文件里无序堆叠修改。
- **[不引入新控件库导致原生感有限]** → 先完成 grouped layout、状态色和控件语义统一，若后续仍不满足，再评估自定义 select/number input 精修。
- **[窗口层改动过大可能影响跨平台或行为稳定性]** → 首轮限制在“保留原生装饰 + 轻量配置调整”，不做高风险窗口框架改造。
