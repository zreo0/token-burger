## Context

当前 Popup 是一个 360x480 的 Tauri 菜单栏小窗，结构为顶部时间范围切换、中部 Burger 可视化、底部总量/花费汇总，并在冷启动时插入状态横幅。现有设计的问题不在于数据获取，而在于视觉语义和信息层级：Burger 结构偏插画式，四类 token 都通过渐变矩形线性映射高度，层内文字会随高度被挤压，Total/Cost 被放在底部，Top Models 没有展示，导致用户无法在打开小窗后快速理解“今天消耗了多少”“缓存是否发挥作用”“主要是哪个模型在消耗”。

这个改动仍然受菜单栏场景约束：窗口可以适度放大，但仍需一眼读完、失焦即走、不能演化成完整仪表盘。与此同时，TokenBurger 的主题表达不能丢失，因此不能简单退化为通用条形图，而需要保留“自下而上组装汉堡”的核心隐喻。

## Goals / Non-Goals

**Goals:**
- 将 Popup 重新组织为更适合菜单栏小窗的层级：时间范围、总量/花费摘要、Burger 主可视化、Top Models、冷启动状态。
- 将 Burger 视觉语义稳定为四层：Input 底部面包、Cache Create、Cache Read、Output 顶部面包。
- 采用层内文字 + 非线性厚度映射，使中间缓存层成为主要的数据对比对象，同时保持所有层可读。
- 为 Today 实时更新与 7D/30D 切换定义两套不同的动效节奏。
- 在不改变现有数据契约的前提下，增加 Top Models 轻量榜单展示。

**Non-Goals:**
- 不新增后端统计字段，不修改 `TokenSummary` 数据结构。
- 不在本次改动中加入趋势图、深度 drill-down 或复杂交互面板。
- 不引入新的 UI 库或设计系统依赖。
- 不修改 settings 页面或 tray 交互逻辑。

## Decisions

### 1. Popup 采用“摘要在上，主可视化居中，补充信息在下”的菜单栏布局
- **Decision**: 将总 Token 与预估花费提升到 Burger 之前，作为打开小窗后的第一阅读层；Top Models 放在 Burger 之后，冷启动状态放在最底部作为轻量状态区。
- **Why**: 用户打开小窗首先关心当前范围的总体消耗，其次才是结构和来源。这样更符合菜单栏工具“先结论、后解释”的阅读路径。
- **Alternative considered**: 保持现有 footer 汇总；但这会继续让关键数据被弱化。

### 2. Burger 改为四层稳定语义，而不是“面包 + 四个数据层 + 面包”
- **Decision**: 将 Input 直接定义为底部面包，将 Output 直接定义为顶部面包；中间两层依次表示 Cache Create 和 Cache Read。
- **Why**: 这样更贴合 TokenBurger 的主题，也能把缓存参与度聚焦在中间两层的“丰满程度”上，强化“缓存越厚，汉堡越饱满”的视觉语言。
- **Alternative considered**: 保留独立上下两片面包再叠四类 token；但这会让层级过多，小窗更拥挤，也削弱输入/输出作为结构锚点的意义。

### 3. 厚度采用文本行高驱动的非线性映射
- **Decision**:
  - Input / Output 视觉高度固定在约 1.5 行文本高度。
  - Cache Create / Cache Read 的厚度范围限定为 1 到 3 行文本高度。
  - 厚度映射采用非线性增长，小值变化更敏感，大值快速逼近上限。
- **Why**: 小窗中最重要的是稳定和可读。线性映射会让极大值撑坏结构、极小值几乎不可见，而文本行高驱动能直接和层内文字排版对齐。
- **Alternative considered**: 继续沿用基于最大值的线性高度计算；但这会导致不同层之间的可比性过于依赖极值，且层内文字容易被压缩。

### 4. 数值直接写在层内，不再额外做外部图例
- **Decision**: 每层直接展示名称与数值，如 `Cache Read 82.1K`，数值与层体合一。
- **Why**: 这能缩短阅读路径，避免在小窗中引入左右对照关系，也更符合“每层自己就是数据本体”的表达方式。
- **Alternative considered**: 将数字放在侧边 callout；但会挤占横向空间，也会削弱汉堡一体化视觉。

### 5. 动画分为“实时更新”与“时间范围切换”两种节奏
- **Decision**:
  - Today 实时更新使用细致、连续、轻量的数字与厚度跟随动画。
  - 7D/30D 切换使用快速、干脆的整体重组动画。
- **Why**: 两种场景的用户心理预期不同。实时更新需要不打扰操作，范围切换需要立即反馈。
- **Alternative considered**: 统一使用同一套 spring 过渡；但会让实时更新显得频繁扰动，让范围切换显得拖沓。

### 6. Top Models 作为补充来源视图保留在 Popup 中
- **Decision**: 使用 `summary.by_model` 在 Burger 下方展示 Top 2 models 的 token 消耗。
- **Why**: Burger 负责展示结构，Top Models 负责展示来源。这个信息维度在当前数据契约下可直接支持，且能为后续扩展统计区预留布局模式。
- **Alternative considered**: 不展示 model 榜单；但这会浪费已有统计能力，也不能满足页面原本设计意图。

### 7. Popup 窗口适度放大，但保持菜单栏工具尺度
- **Decision**: 将窗口尺寸从 360x480 提升到约 390x540（允许实现时在 400x560 内微调）。
- **Why**: 需要给 Burger、Top Models 和状态区更多呼吸空间，但仍需维持菜单栏弹窗的紧凑感。
- **Alternative considered**: 保持原尺寸；但在增加 Top Models 后内容会明显拥挤。

## Risks / Trade-offs

- **[风险] 中间缓存层数据很小时，四层看起来差异不明显** → **Mitigation**: 使用非线性映射，让低值区间的厚度变化更敏感。
- **[风险] 将 Input/Output 固定为面包会减弱其绝对量级比较** → **Mitigation**: 直接在层内显示精确数值，并将总 Token 与花费放在摘要区强化全局认知。
- **[风险] 层内文字叠加在不同背景色上可能影响可读性** → **Mitigation**: 使用克制的语义色与高对比文本，避免高饱和渐变和复杂纹理。
- **[风险] Popup 加大后可能削弱菜单栏小窗气质** → **Mitigation**: 控制在 390x540 左右，并保持单屏阅读完成，不引入滚动和深层交互。
- **[风险] Top Models 增加后页面信息过多** → **Mitigation**: 只展示 Top 2，采用轻量列表，不附加复杂图形。

## Migration Plan

1. 先调整 `burger-ui` 规格，明确新的 Popup 布局、层级语义和动画节奏。
2. 再按规格改造 Popup 页面结构与 Burger 组件实现。
3. 调整窗口尺寸配置与样式细节，确保小窗在浅色/深色模式下都保持稳定。
4. 最后补充或更新相关前端测试，验证时间范围切换、实时更新和 Top Models 展示。

## Open Questions

- Top Models 是否需要在 token 数值之外额外展示占比；当前先不要求，只保留名称与数值。
- 当缓存层为 0 时是否保留极细骨架；当前设计倾向保留最薄层，以维持结构完整性。
